# ADR-001: Container Host Architecture for HLH

**Status:** Adopted
**Date:** 2026-05-29
**Author:** Kevin
**System:** HLH (Home Lab Hardware)
**Platform:** Proxmox 9.2 on prox01 (192.168.1.10)

## Context

The homelab requires a lightweight, reproducible, secure container host for running Docker-based applications. The host must be provisioned via OpenTofu (HCL) and configured via Ansible, with persistent data stored on the Proxmox node's ZFS storage pool. The environment must support GUI and TUI management tools and allow each application to receive its own dedicated IP via VLAN-aware networking.

## Decision

Deploy a Docker container host inside an **unprivileged LXC** on Proxmox, using:

- **Docker Engine** for container runtime
- **Dockhand** for GUI management (image: `fnsys/dockhand`)
- **LazyDocker** for TUI management
- **ZFS bind-mounts** for persistent storage
- **VLAN-aware networking** for per-app IP assignment
- **4 vCPU / 4GB RAM / 32GB rootfs** initial resource allocation
- **OS:** Ubuntu 26.04 LTS

The LXC will be **vmid 102**, provisioned using OpenTofu and configured using Ansible.

## Storage

- **Rootfs:** 32GB subvolume on `RaidZ1-6TB`
- **Docker data:** `/var/lib/docker` → `RaidZ1-6TB/hlh-docker/docker-data` (ZFS bind-mount)
- **Dockhand data:** `/srv/dockhand/data` → `RaidZ1-6TB/hlh-docker/dockhand-data` (ZFS bind-mount)
- **ZFS pool:** `RaidZ1-6TB` (NOT `rpool`)

Bind-mounts are configured in the OpenTofu resource via `mount_block`. The datasets must exist
on the Proxmox host before `tofu apply` — see ADR-002 for the storage strategy.

### ZFS Dataset Inventory

| Dataset | Mount Point in LXC | Purpose |
|---|---|---|
| `RaidZ1-6TB/hlh-docker/docker-data` | `/var/lib/docker` | Docker images, layers, named volumes |
| `RaidZ1-6TB/hlh-docker/dockhand-data` | `/srv/dockhand/data` | Dockhand container metadata |
| `RaidZ1-6TB/hlh-docker/rootfs` | *(unused)* | Reserved, not currently mounted |

## Network

- **VLAN-aware bridge** (tagged interfaces reserved for future use)
- **Current subnet:** 192.168.1.0/24
- **Docker LXC IP:** 192.168.1.13
- **Per-app IP assignment:** Via VLAN segmentation when needed

## Rationale

- **Unprivileged LXC** provides strong security isolation without GPU/kernel access
- **Docker + Dockhand** gives modern container management
- Dockhand image (`fnsys/dockhand`) is available on Docker Hub
- **ZFS bind-mounts** preserve state across LXC rebuilds
- **VLAN-aware networking** enables future app segmentation without reverse proxies
- **OpenTofu + Ansible** enables reproducible, idempotent provisioning

## Consequences

### Positive
- Strong security boundary (unprivileged LXC)
- Stateless compute, stateful storage on ZFS
- Simple reprovisioning (destroy/recreate LXC without data loss)
- Clean per-app networking via VLANs
- Easy automation with IaC
- Dockhand is lighter than Portainer with better UX

### Negative
- Dockhand is newer and less battle-tested than Portainer
- Docker inside unprivileged LXC requires `nesting` + `keyctl` tuning
- macvlan/ipvlan networking can complicate container-to-host communication

## Alternatives Considered

|| Alternative | Reason Rejected |
||---|---|
|| Privileged LXC | Weak security boundary; no GPU needed |
|| VM instead of LXC | Higher overhead, slower provisioning |
|| ZFS passthrough inside LXC | Complexity, nested ZFS performance penalty |
|| Portainer | Dockhand preferred; lighter footprint, modern UX |
|| Reverse proxy model | VLAN-aware per-app IPs preferred for simplicity |

## Decision Outcome

This architecture provides the best balance of:
- Security (unprivileged LXC)
- Reproducibility (stateless LXC + stateful ZFS)
- Operational simplicity (Dockhand + LazyDocker)
- IaC automation (OpenTofu + Ansible)
- Future extensibility (VLAN-aware networking)
