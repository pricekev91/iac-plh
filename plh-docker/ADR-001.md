# ADR-001: Container Host Architecture for PLH

**Status:** Adopted
**Date:** 2026-06-24
**Author:** Kevin
**System:** PLH (Personal Lab Hardware)
**Platform:** CachyOS (Arch Linux derivative) with LXD 6.9

## Context

The personal lab requires a lightweight, reproducible, secure container host for running Docker-based applications. The host must be provisioned via LXD (Linux Containers Daemon) on a CachyOS desktop — pure bash scripts, no Terraform, no Ansible, no OpenTofu. The environment must support GUI and TUI management tools (Dockhand, LazyDocker) with persistent data stored on the host's ZFS storage pool.

## Decision

Deploy a Docker container host inside an **unprivileged LXD container** on CachyOS, using:

- **Docker Engine** for container runtime
- **Dockhand** for GUI management (image: `fnsys/dockhand`)
- **LazyDocker** for TUI management
- **ZFS bind-mounts** for persistent storage
- **LXD** (not Proxmox) for container lifecycle management
- **4 vCPU / 4GB RAM / 32GB rootfs** initial resource allocation
- **OS:** Ubuntu image from LXD image remote (ubuntu:latest)

The container will be named `plh-docker`, provisioned using `lxc launch` and configured using pure bash scripts.

## Storage

- **Rootfs:** 32GB subvolume managed by LXD (ZFS-backed on CachyOS)
- **Docker data:** `/var/lib/docker` → `/srv/data/docker` (bind-mounted from host)
- **Dockhand data:** `/srv/dockhand/data` → `/srv/data/dockhand/data` (bind-mounted from host)
- **LXD storage:** Default pool (ZFS, created by `lxd init`)

Bind-mounts are configured via `lxc config device add`. The directories must exist
on the CachyOS host before the container is created.

### Host Directory Structure

| Host Path | Container Path | Purpose |
|---|---|---|
| `/srv/data/docker` | `/srv/data/docker` | Docker images, layers, named volumes |
| `/srv/data/dockhand/data` | `/srv/data/dockhand/data` | Dockhand container metadata |
| `/srv/data/dockhand/run` | `/srv/data/dockhand/run` | Dockhand runtime files |

## Network

- **Default LXD bridge** (lxdbr0)
- **Container networking:** NAT by default, port forwarding for Dockhand (port 80)
- **Dockhand GUI:** Exposed via `-p 80:3000` on the host interface
- **Docker-in-LXC:** Docker daemon runs inside the container, connecting to containers on the container's internal network

## Rationale

- **LXD over Proxmox:** CachyOS is a desktop OS — LXD is the native container runtime. Proxmox is a full hypervisor designed for dedicated server hardware, not desktop workstations.
- **Unprivileged LXD** provides strong security isolation without GPU/kernel access
- **Docker + Dockhand** gives modern container management
- Dockhand image (`fnsys/dockhand`) is available on Docker Hub
- **ZFS bind-mounts** preserve state across container rebuilds
- **LXD** is simpler to manage on a desktop than Proxmox — single daemon, `lxc` CLI, no web admin panel overhead
- **Pure bash** eliminates dependencies on Terraform, Ansible, or OpenTofu

## Consequences

### Positive
- Strong security boundary (unprivileged LXC)
- Stateless compute, stateful storage via bind-mounts
- Simple reprovisioning (destroy/recreate container without data loss)
- Clean automation with bash scripts
- Dockhand is lighter than Portainer with better UX
- No Proxmox dependency — works on CachyOS desktop natively

### Negative
- Dockhand is newer and less battle-tested than Portainer
- Docker inside unprivileged LXC requires `security.nesting=true` tuning
- LXD networking is simpler than Proxmox — no built-in VLAN management
- Single-machine setup (no clustering)

## Alternatives Considered

| Alternative | Reason Rejected |
|---|---|
| Proxmox VE | Overkill for desktop; CachyOS is not a server OS |
| Privileged LXD | Weak security boundary; no GPU needed |
| VM instead of LXD | Higher overhead, slower provisioning |
| Portainer | Dockhand preferred; lighter footprint, modern UX |
| Ansible / Terraform | Bash scripts are simpler for single-machine setup |
| Docker native (no container) | Less isolation; LXD provides security boundary |

## Decision Outcome

This architecture provides the best balance of:
- Security (unprivileged LXC)
- Simplicity (LXD + bash, no Proxmox)
- Operational simplicity (Dockhand + LazyDocker)
- Reproducibility (stateless container + stateful bind-mounts)
